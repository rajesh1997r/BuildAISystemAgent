1. Executive Summary

This report describes the design, implementation, and evaluation of a multi-agent AI content creation system built using the CrewAI framework and OpenAI's GPT-4o-mini model. The system accepts a user-supplied topic and autonomously produces a publication-ready Markdown article by coordinating three specialized AI agents through a sequential pipeline.

A central ContentCreationController orchestrates the pipeline. The Idea Generator Agent researches trending angles and selects the most compelling idea. The Content Writer Agent drafts a structured 400–700 word article. The Content Editor Agent refines grammar, tone, and structure before producing the final output. Four custom-built tools support the agents: a web search tool, a text processing tool, an output formatter, and a content quality analyzer — the last of which is both an agent-callable tool and a post-pipeline evaluator.

The system includes comprehensive input validation, exponential-backoff retry logic, a shared memory layer for state persistence, evaluation metrics, and a full unit test suite — all without any additional external API beyond OpenAI.

2.2 Data Flow

The user invokes main.py via the command line with an optional topic string. After input validation, a SharedMemory instance is created and handed to the ContentCreationController. The controller builds three interdependent CrewAI Tasks and assembles them into a Crew running with Process.sequential. Each task's output is automatically forwarded to the next via CrewAI's context parameter. Once the crew completes, the controller extracts the final Markdown content from the CrewOutput.raw attribute, persists it to shared memory, and returns the result to main.py. The content is then scored by the ContentQualityAnalyzer and wrapped in EvaluationMetrics before being printed to the console.

3. Agent Roles and Responsibilities

All three agents are created via factory functions (e.g., create_idea_generator_agent()) that return a configured crewai.Agent instance. Each agent has allow_delegation=False and max_retry_limit=2. An optional llm parameter allows the LLM model to be overridden, which is useful for testing with cheaper models.

Agent Role Name Primary Goal Tools Assigned
Idea Generator Creative Content Idea Generator Research current trends and brainstorm 5–10 specific content angles. End output with a SELECTED IDEA: line. WebSearchTool, TextProcessingTool
Content Writer Professional Content Writer Transform the selected idea into a 400–700 word Markdown article with title, introduction, 3+ body sections, bullet points, and conclusion. WebSearchTool, OutputFormatterTool
Content Editor Expert Content Editor Fix grammar/spelling, replace passive voice, shorten long sentences, self-check quality score (target ≥ 8), and produce publication-ready Markdown. TextProcessingTool, OutputFormatterTool, ContentQualityAnalyzer

3.1 Agent Communication

Agents communicate through CrewAI's task context chaining. When writer_task is defined with context=[idea_task], CrewAI automatically appends the Idea Generator's full output to the Writer's prompt at runtime. The same pattern connects the Writer's output to the Editor. This eliminates the need for explicit message-passing code — the framework manages the information handoff.

The SharedMemory class provides a secondary communication layer at the controller level, storing topic, status, draft_content, and final_content as the pipeline progresses. This enables the controller to recover partial results if a retry is needed.

3.2 Prompting Strategy

Each agent is given a detailed backstory string that is injected into the LLM's system prompt. For example, the Idea Generator's backstory describes it as "an award-winning content strategist with a decade of experience in SEO and audience engagement," which consistently biases the model toward producing specific, trend-aware ideas rather than generic outlines. Task descriptions include explicit output format requirements (e.g., "end with exactly SELECTED IDEA: <one sentence>") to make downstream parsing reliable.

4. Tool Integration and Functionality

All tools subclass crewai.tools.BaseTool and define a Pydantic input schema for automatic type validation. CrewAI serializes tool calls to JSON, validates them against the schema, and invokes the tool's _run() method. Each tool handles invalid inputs and network errors gracefully, returning descriptive error strings rather than raising exceptions that would crash the agent loop.

5. Custom Tool Documentation — Content Quality Analyzer

5.1 Overview

The ContentQualityAnalyzer (tools/content_quality_analyzer.py) is the project's mandatory custom tool. It evaluates generated content using rule-based heuristics and returns a numeric quality score from 1 to 10 along with actionable improvement suggestions. It is uniquely dual-purpose in this system:

  • As a CrewAI BaseTool — the Content Editor Agent can call it mid-task to self-check its draft before finalizing, implementing an in-loop quality feedback cycle.
  • As a plain Python class — main.py calls analyzer.analyze(final_content) directly after the pipeline to produce the post-pipeline quality report shown to the user.

5.2 Inputs and Outputs

Usage ModeInputOutput
Agent tool call (_run) content: str — the article text Formatted string: score, visual bar chart breakdown, bullet-point suggestions
Direct Python (analyze) content: Any — validated internally Dict: {"score": float, "suggestions": list, "breakdown": dict}

5.3 Scoring Algorithm

Four sub-scores are computed independently and combined using a weighted average:

DimensionWeightWhat Is MeasuredMethod
Length20% Word count relative to ideal range (400–1500 words) _analyze_length()
Readability30% Average sentence length (≤ 20 words target) and fraction of sentences exceeding 35 words _analyze_readability()
Engagement30% Presence of question marks, Markdown headings (≥ 3), bullet lists, bold emphasis _analyze_engagement()
Structure20% Paragraph count (≥ 5 for full marks), conclusion keywords, introduction keywords _analyze_structure()

The final score is clamped to [1.0, 10.0]. If the analyzer detects no issues, it returns the message "Content quality looks great!" If the input is not a string, a type-error message is returned with a score of 0 rather than raising an exception.

5.4 Integration with Editor Agent

The Content Editor Agent's task description explicitly instructs it to call the Content Quality Analyzer Tool and aim for a score of 8 or above before proceeding to the Output Formatter Tool. This creates a self-improvement feedback loop: the agent can read the suggestions returned by the analyzer and apply them to its own draft within the same task execution.

5.5 Limitations

  • The scoring is heuristic, not semantic — it cannot detect factual inaccuracies or assess argument quality.
  • Readability scoring uses word count per sentence, not a standard formula like Flesch-Kincaid.
  • Keyword detection for conclusions and introductions relies on a fixed list and may miss creative phrasing.

6. Orchestration and Memory Management

6.1 Task Orchestration

The ContentCreationController._build_tasks() method constructs three crewai.Task objects. Sequential execution is enforced by passing Process.sequential to the Crew constructor. This design is intentional: the Writer cannot begin until the Idea Generator has committed to a SELECTED IDEA, and the Editor cannot begin until the Writer has produced a complete draft. A parallel process would not make logical sense here.

Agent communication across tasks is implemented via the context parameter. writer_task has context=[idea_task] and editor_task has context=[writer_task]. CrewAI appends the full text output of each context task to the next agent's prompt automatically.

6.2 Retry and Error Recovery

The ContentCreationController.run() method wraps the pipeline in a retry loop with a configurable max_retries parameter (default: 2). On each failure, the system waits 2attempt seconds (exponential backoff) before retrying. If all retries fail, the controller returns any partial content already stored in SharedMemory so the user receives the best available output rather than an empty response. Each agent also has its own max_retry_limit=2 at the CrewAI level for transient LLM API errors.

6.3 Shared Memory

The SharedMemory class (utils/memory.py) is a thread-safe dictionary wrapper that persists workflow state across the entire pipeline. It stores the following keys:

KeySet ByPurpose
topicController at startThe user's topic string
statusController throughoutrunning / completed / error
selected_ideamain.py after pipelineThe idea chosen by the Idea Generator
final_contentController after pipelineThe polished Markdown article
errorController on failureLast exception message

Every set() and delete() call is logged to an internal _history list with an ISO 8601 timestamp, providing a full audit trail of state changes. The to_json() method exports the store for debugging.

7.3 Evaluation Metrics

The EvaluationMetrics dataclass (utils/metrics.py) tracks three values for every pipeline run. A run is marked acceptable if all three thresholds are met:

  • Content length > 100 characters
  • Quality score ≥ 5.0 / 10
  • Response time < 300 seconds

8. Challenges and Solutions

ChallengeSolution
CrewAI imports OpenAI eagerly — importing agent modules before the API key is loaded caused crashes. Moved load_dotenv() and the API key check to the very top of main.py, before any project imports. The program exits with a friendly error if the key is absent.
CrewOutput format changed across CrewAI versions — accessing the final content string broke between versions 0.51 and later releases. Implemented the _extract_content() helper in the controller that checks for .raw, then .output, then falls back to str(). This makes the system forward-compatible.
DuckDuckGo rate limiting and network failures in test environments caused the web search tool to raise exceptions mid-pipeline. Wrapped the DDGS call in a try/except. On any failure, the tool falls back to _mock_search() which returns representative placeholder results, allowing the pipeline to complete offline.
Pydantic v2 conflict with class-level tuning constants in ContentQualityAnalyzer after converting it to a BaseTool — Pydantic treated integer literals as model fields. Annotated all tuning constants with ClassVar[int] and ClassVar[float], which instructs Pydantic v2 to exclude them from model field processing.
Reliable idea extraction — the Idea Generator sometimes buried the selected idea in the middle of its output rather than on the last line. The task description explicitly requires the phrase SELECTED IDEA: on the very last line. The Writer agent's prompt reinforces this by instructing it to "use the SELECTED IDEA from the previous task," making the instruction robust against formatting variation.
Empty input reaching agents — early versions let blank topics propagate all the way to CrewAI, resulting in generic or nonsensical output. Added a two-stage input guard in run_content_creation(): empty strings and topics under 3 characters are rejected immediately, before any agent or memory object is created, returning a structured error dictionary.

9.2 Strengths

  • Fully modular architecture — agents, tools, and utilities are independently testable and replaceable.
  • Offline-capable — mock search fallback and unit tests require no API key or network access.
  • Robust error handling at every layer: input validation, tool-level try/except, agent retry limits, and controller-level exponential backoff.
  • The custom tool (ContentQualityAnalyzer) is genuinely dual-use — it drives agent behavior during the pipeline and provides user-facing evaluation after it.
  • Clean CLI interface with three modes: default demo, custom topic, and batch test runner.

9.3 Limitations and Future Improvements

  • No persistent long-term memory. SharedMemory is in-memory and resets each run. A future version could persist to SQLite or a vector store so the system learns from previous articles (e.g., avoid repeating topics, build on established style).
  • Sequential-only execution. The Idea Generator could run multiple search queries in parallel using CrewAI's Process.hierarchical mode or async tool calls. This could reduce response time by 30–40%.
  • Heuristic quality scoring. The ContentQualityAnalyzer uses rule-based heuristics rather than semantic analysis. Integrating a readability library (e.g., textstat) or an LLM-based critique would produce more nuanced scores.
  • Single output format. The system currently targets Markdown. A future extension could add a formatting selector that lets the user request HTML, plain text, or a specific word count at the CLI level.
  • No reinforcement learning loop. Agent prompts are static. A future version could use the quality score as a reward signal to iteratively refine prompts via a meta-agent or automated prompt optimization framework such as DSPy.

10. Conclusion

This project demonstrates a complete, production-quality multi-agent AI system built on CrewAI that fulfills all assignment requirements: a controller agent, three specialized agents, three built-in tools, one custom tool with agent integration, sequential orchestration, shared memory, and a comprehensive test suite.

The most significant design decision was making the ContentQualityAnalyzer both an agent-callable tool and a post-pipeline evaluator. This means quality feedback operates at two levels — the Editor Agent can self-correct mid-task, and the user receives a full scored breakdown after the pipeline completes.

The system handles all three required test scenarios correctly: a well-defined topic produces a high-scoring article, a vague topic still results in a usable article, and empty input is rejected gracefully before any API call is made. The use of exponential backoff retries, offline-capable tools, and structured error returns ensures the system is robust in real-world conditions, not just ideal test cases.

Future development would focus on persistent memory, parallel task execution, and semantic quality scoring to move the system closer to a fully autonomous publishing assistant.
